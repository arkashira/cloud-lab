## 1. Acceptance Criteria
- **AC1 – Register success**: `POST /auth/register` with a unique email and valid password returns **201** and a JSON body containing a JWT (`token`) that can be decoded with the configured `JWT_SECRET`.
- **AC2 – Register duplicate**: Registering with an email that already exists returns **409 Conflict** and an error message `"Email already in use"`.
- **AC3 – Login success**: `POST /auth/login` with correct email/password returns **200** and a JWT (`token`) that matches the user’s id when decoded.
- **AC4 – Login failure**: Invalid email **or** password returns **401 Unauthorized** with a generic error `"Invalid credentials"` (no indication which field failed).
- **AC5 – Password‑reset request**: `POST /auth/reset-password` with any email returns **200** and a generic success message `"If the email exists, a reset link has been sent"`; the response does not reveal existence of the address.
- **AC6 – JWT configuration**: The token’s expiration respects `process.env.JWT_EXPIRES_IN` (default `1h`). If `process.env.JWT_SECRET` is missing or equals the default placeholder, the server logs a warning in development mode but still issues a token.
- **AC7 – Model abstraction**: The controller uses only the exported methods `User.findByEmail(email)` and `User.create(payload)`. Swapping the underlying ORM (e.g., Mongoose → Prisma) does not require changes to the controller code.
