## Unified Real‑Time Collaboration Stack  
**Goal:** Give you a single, production‑ready implementation that (1) runs a WebSocket‑compatible server alongside an existing Express/Node app, (2) tracks who is editing which document, (3) broadcasts document changes **and** connection‑state events, (4) offers a clean, reusable front‑end API (both a plain‑JS utility and a React hook), and (5) stays easy to test and extend.

Below you will find:

| Piece | What it does | Why we kept it |
|------|---------------|----------------|
| **Server** (`backend/websockets/server.js`) | • Handles HTTP upgrade to WebSocket only on `/ws` <br>• Uses the lightweight **`ws`** library (no extra dependencies) <br>• Keeps per‑document editor sets (`editingUsers`) <br>• Emits generic **connection** events (`user‑connected`, `user‑disconnected`) for UI‑level presence <br>• Broadcasts document‑specific `update` messages and a `user_list` payload | Combines Candidate 1’s fine‑grained document handling with Candidate 2’s global presence events. `ws` is preferred for low‑overhead servers; we still expose the same events Socket.io would have emitted, but in a format that works with any client. |
| **Exported `start(port)`** | Allows the main app to import the module and call `start(8080)` (or any port) – no need to run a separate process. | Keeps deployment simple (single Node process). |
| **Front‑end utility** (`frontend/src/utils/websocketClient.js`) | • Simple wrapper around the browser `WebSocket` API <br>• `connect({docId, username, host})` returns the raw socket <br>• `on(event, cb)` registers callbacks for `update`, `user_list`, `user-connected`, `user-disconnected` <br>• `sendUpdate(changes)` and `sendMessage(payload)` for generic messages <br>• `disconnect()` cleans up | Merges Candidate 1’s typed listeners with Candidate 2’s generic “message” handling, giving you both high‑level (document) and low‑level (presence) events. |
| **React hook** (`frontend/src/hooks/useWebSocket.js`) | • `useWebSocket({docId, username, host})` returns `{socket, sendUpdate, sendMessage, users, connectionCount}` <br>• Internally uses the utility above, so the hook stays tiny and testable <br>• Auto‑reconnect on URL change, cleans up on unmount | Provides the ergonomics developers expect in a modern React code‑base while re‑using the same underlying logic. |
| **Testing helpers** | The server module also exports the raw `wss` instance (`_wss`) so unit‑tests can inject mock sockets. | From Candidate 1 – useful for CI. |

## 1️⃣ Backend – `backend/websockets/server.js`

```javascript
// /opt/axentx/cloud-lab/backend/websockets/server.js
/**
 * Real‑time collaboration server.
 *
 * Features
 * --------
 *  - Upgrade only `/ws` requests to a WebSocket (so the same HTTP server can serve
 *    your existing Express routes).
 *  - Per‑document editor tracking (`editingUsers`).
 *  - Global presence events (`user-connected`, `user-disconnected`).
 *  - Simple JSON protocol:
 *        {type:'update',   changes:{...}}          // document delta
 *        {type:'user_list',users:['alice','bob']} // refreshed editor list
 *        {type:'message',  payload:{...}}         // any custom payload
 *        {type:'user-connected', userId, count}
 *        {type:'user-disconnected', userId, count}
 *
 *  - In‑memory state only – plug in Redis or a DB later if you need persistence.
 */

const http = require('http');
const WebSocket = require('ws');
const url = require('url');

// ---------------------------------------------------------------------------
// 1️⃣  HTTP server – can be imported by the main Express app
// ---------------------------------------------------------------------------
const server = http.createServer();               // no request handler needed here
const wss = new WebSocket.Server({ noServer: true });
