# /opt/axentx/cloud‑lab/docs/terraform_guide.md  
# Terraform Usage Guide for the Cloud‑Lab Sandbox

> **TL;DR** – Create a folder, write a `main.tf`, run `terraform init → plan → apply`, and destroy when you’re done.

---

## 1. Introduction  

This guide walks you through provisioning and tearing down AWS resources in the Cloud‑Lab sandbox using Terraform. All commands run from a Linux terminal inside the sandbox.

---

## 2. Prerequisites  

| Item | Why it matters | How to verify |
|------|----------------|---------------|
| **Sandbox access** | You need a user account with SSH or web‑terminal access. | `whoami` |
| **Terraform knowledge** | Basic HCL syntax and the Terraform workflow. | `terraform --help` |
| **AWS credentials** | Terraform must authenticate to AWS. | `aws sts get-caller-identity` (or `echo $AWS_ACCESS_KEY_ID`) |

> **Tip** – The sandbox comes with a pre‑configured `~/.aws/credentials` file for the `sandbox` profile. If you prefer environment variables, export them before running Terraform.

---

## 3. Installation & Verification  

Terraform is pre‑installed. Confirm the version:

```bash
terraform --version
```

> **Expected output** – e.g., `Terraform v1.8.5`.  
> If you see an older version, run `sudo apt-get update && sudo apt-get install terraform`.

---

## 4. Create a Working Directory  

```bash
mkdir -p ~/terraform-configs
cd ~/terraform-configs
```

All subsequent commands run from this directory.

---

## 5. Write Your First Configuration  

Create `main.tf`:

```hcl
terraform {
  required_version = ">= 1.0"
  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  # Remote state is optional but recommended for team use
  backend "local" {
    path = "terraform.tfstate"
  }
}

provider "aws" {
  region = "us-west-2"
}

resource "aws_instance" "example" {
  ami           = "ami-0c55b159cbfafe1f0"  # Amazon Linux 2
  instance_type = "t2.micro"

  tags = {
    Name = "cloud-lab-demo"
  }
}
```

> **Why this layout?**  
> * `terraform {}` block declares provider and backend.  
> * `required_version` protects against breaking changes.  
> * `backend "local"` keeps state in the sandbox; switch to S3/Consul for multi‑user setups.

---

## 6. Initialise Terraform  

```bash
terraform init
```

* Downloads provider plugins.  
* Sets up the backend.  
* Prints a success message.

---

## 7. Review the Execution Plan  

```bash
terraform plan
```

* Shows what Terraform will create/modify.  
* Look for `+ aws_instance.example` – a new instance will be created.  
* If you see `-` or `~`, Terraform will delete or modify resources.

---

## 8. Apply the Configuration  

```bash
terraform apply
```

* Terraform will prompt for confirmation (`Do you want to perform these actions? (yes/no)`).  
* Type `yes` to proceed.  
* After completion, you’ll see the instance ID and public IP.

---

## 9. Inspect the State  

```bash
terraform show
```

* Displays the current state in a human‑readable format.  
* Useful for debugging or verifying resource attributes.

---

## 10. Clean Up  

```bash
terraform destroy
```

* Prompts for confirmation.  
* Removes all resources created by this configuration.  
* Run this whenever you’re finished to avoid unnecessary charges.

---

## 11. Best Practices & Tips  

| Practice | Why it matters | How to implement |
|----------|----------------|------------------|
| **Use modules** | Keeps code DRY and reusable. | `module "ec2" { source = "./modules/ec2" }` |
| **Sensitive variables** | Prevents secrets from leaking into logs. | `variable "db_password" { sensitive = true }` |
| **Environment variables** | Keeps credentials out of code. | `export TF_VAR_db_password="s3cr3t"` |
| **Version control** | Tracks changes and supports collaboration. | `git init && git add . && git commit -m "Initial Terraform"` |
| **Regular updates** | Leverages bug fixes and new features. | `terraform init -upgrade` |
| **Remote state** | Enables team colla
