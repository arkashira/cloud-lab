## 1. Acceptance Criteria
- **AC‑1** – Every AWS SDK error response received during account creation is written to `/opt/axentx/cloud‑lab/backend/logs/aws_errors.log` with a JSON line containing **timestamp (ISO‑8601), request_id, raw_error_payload**.  
- **AC‑2** – The error‑classification service maps a known error code (e.g., `83730`) to a taxonomy entry **within 2 seconds** of the error being logged (measured by a monotonic timer in unit tests).  
- **AC‑3** – For a known error, the user session view displays a **human‑readable code** (e.g., `ACCOUNT_LIMIT_EXCEEDED`) and a **description** pulled from the taxonomy.  
- **AC‑4** – Errors that do not match any taxonomy entry are persisted to the **`pending‑review` S3 bucket** (or local mock bucket) with the original payload and a flag `needs_review: true`.  
- **AC‑5** – The classification function is **pure** (no side‑effects other than logging/persisting) and returns a deterministic `ClassificationResult` object containing `code`, `description`, `is_known`, and `processing_time_ms`.  
- **AC‑6** – The logging subsystem rotates the file when it exceeds **10 MB**, preserving older logs with a timestamped suffix.  
- **AC‑7** – All failures in the classification pipeline (e.g., logging I/O error, S3 upload failure) are caught, logged as `CRITICAL`, and do not crash the account‑creation workflow.
