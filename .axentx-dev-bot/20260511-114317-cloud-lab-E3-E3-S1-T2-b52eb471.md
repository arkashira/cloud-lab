Below is a single, self‑contained implementation that

1. **Keeps the clean separation between “what” (the *runner*) and “how” (the *executor*).**  
2. **Supports both a mocked executor (for unit tests / local demos) and a real AWS executor** that talks to the EKS API.

> Call `SubmitLab` → get a `labID`.  
> Call `GetKubeconfig(labID)` → kube‑config string.  
> Call `DeleteLab(labID)` or let the background goroutine delete it after the TTL.

import (

}

- **AC1** – **API submission**: POST `/labs/eks` with a minimal EKS manifest returns HTTP 201 and a JSON payload containing a non‑empty `labId` (UUID format) within **2 seconds**.  
- **AC2** – **Cluster provisioning time**: The created EKS cluster reaches `ACTIVE` state and a valid `kubeconfig` is stored **≤ 4 minutes** after the request is accepted.  
- **AC3** – **Kubeconfig validity**: The returned `kubeconfig` can be parsed by `client-go`/`kubectl` (i.e., contains required fields: `apiVersion`, `clusters`, `contexts`, `users`, and a valid certificate data block).  
- **AC4** – **Automatic teardown**: A background job deletes the cluster **≤ 5 minutes** after the 24‑hour TTL expires, and the lab record is marked `DESTROYED`.  
- **AC5** – **Manual teardown**: DELETE `/labs/{labId}` returns HTTP 202, triggers immediate cluster deletion, and the lab status transitions to `DESTROYED` within **2 minutes**.  
- **AC6** – **Mock executor behavior**: When the service is configured with `MockEksExecutor`, cluster creation simulates a **3‑second** delay and returns a well‑formed kubeconfig string; deletion simulates a **1‑second** delay.  
- **AC7** – **Error handling**: Invalid manifests (e.g., missing required fields) result in HTTP 400 with a descriptive error; AWS SDK failures result in HTTP 502 with the underlying error logged.
