# cloud‑lab

> **Power developers with scalable cloud solutions.**  
> cloud‑lab is a lightweight, opinionated platform that lets you spin up, configure, and tear down cloud‑native applications with a single command‑line workflow.  
> The stack is intentionally minimal – Terraform provisions the infrastructure, Ansible configures the instances, and a small set of helper scripts glue everything together.

---

## Why cloud‑lab?

| Feature | What it gives you |
|---------|-------------------|
| **Scalable** | Uses Terraform modules that are fully idempotent and rely on the latest AWS provider. |
| **Flexible** | Supports any language or framework – you only need to add a new Ansible role or Terraform module. |
| **Secure** | All secrets are passed through the AWS Secrets Manager and the Ansible vault. |
| **Collaborative** | Built‑in GitHub Actions CI that runs linting, unit tests, and a smoke‑test of the deployed stack. |
| **Extensive docs** | Every file is documented in‑line and the README covers the entire workflow. |

---

## Quick‑Start

> **Prerequisites**  
> * Terraform v1.5+ (bundled in the repo, but you can install the latest with `brew install terraform`)  
> * Ansible v2.12+ (`pip install ansible`)  
> * AWS CLI configured with credentials that have `AdministratorAccess` (or the minimal permissions required by the modules).  

```bash
# 1️⃣ Clone the repo
git clone https://github.com/[username]/cloud-lab.git
cd cloud-lab

# 2️⃣ (Optional) Create a virtual environment
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt   # installs ansible

# 3️⃣ Provision the infrastructure in one step
terraform init
terraform apply -auto-approve

# 4️⃣ (Optional) Configure the provisioned resources
ansible-playbook -i inventory/hosts.ini playbooks/deploy.yml

# 5️⃣ To tear down everything
ansible-playbook -i inventory/hosts.ini playbooks/destroy.yml
```

> **Tip** – The `terraform apply` step will create all AWS resources.  
> The Ansible playbooks are idempotent; you can re‑run them safely.  
> The `destroy.yml` playbook cleans up *every* resource created by `deploy.yml`.
