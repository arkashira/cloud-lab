## 1. Acceptance Criteria
- **AC1 – Credential Login Success:** A user with valid cloud‑lab credentials can obtain a JWT access token after Azure AD authentication, and the token contains a non‑expired `exp` claim.
- **AC2 – Role‑Based Azure Access:** The JWT includes a `roles` claim that matches the user’s Azure RBAC assignments; attempts to call an Azure service without the required role return HTTP 403.
- **AC3 – Audit Logging:** Every authentication attempt (success or failure) creates an audit log entry containing `user_id`, `timestamp`, `outcome`, and `source_ip`.
- **AC4 – Token Refresh:** A valid refresh token can be exchanged for a new access token without re‑prompting the user, and the new token’s `iat` is later than the original.
- **AC5 – Multi‑Tenant Isolation:** Users from different Azure tenants receive tokens scoped only to their tenant; cross‑tenant resource access is denied.
- **AC6 – Error Propagation:** Invalid credentials, expired tokens, or mis‑configured Azure AD client result in a standardized error response (`error_code`, `message`) with HTTP 401/403 as appropriate.
- **AC7 – Performance SLA:** Authentication flow (credential submit → token issuance) completes within 800 ms under load of 200 concurrent requests (95th percentile).
